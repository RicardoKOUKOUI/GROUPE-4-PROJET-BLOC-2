01 — Principes directeurs de l'architecture
01
Une responsabilité par fichier
Chaque fichier CSS ne gère qu'une seule section ou un seul type de composant. Si on doit expliquer ce que fait un fichier et que la réponse contient le mot "et", c'est qu'il fait trop de choses.
02
Séparation global / local
Les styles qui s'appliquent à tout le site (reset, couleurs, typographie) vivent dans base.css. Les styles d'une section ne sortent jamais de leur fichier dédié.
03
Composants réutilisables isolés
Les éléments qui apparaissent dans plusieurs sections (boutons, cartes) ont leur propre fichier dans styles/components/. On modifie un bouton à un seul endroit, le changement se répercute partout.
04
Pensée à long terme
La structure anticipe l'arrivée de nouvelles pages (Portfolio, Blog, Espace Client). Chaque nouvelle page crée son propre sous-dossier et réutilise les composants existants sans toucher aux fichiers actuels.

03 — Tableau des fichiers CSS
Fichier Type Responsabilité unique Éléments gérés + Justification
base.css Global Fondations visuelles communes à toutes les pages du site Reset universel (* margin/padding: 0), police globale (Arial), couleur de texte par défaut (#333)
Pourquoi : Ces règles s'appliquent partout. Les centraliser évite de les répéter dans chaque fichier. Si on change la police demain, on touche à un seul endroit.
header.css Section Mise en page et style de la barre de navigation .b1 (barre nav), .b1a (logo), .b1b a (liens nav + hover)
Pourquoi : Le header est présent sur toutes les futures pages. L'isoler permet de le modifier ou de le rendre sticky sans risquer de casser les autres sections.
hero.css Section Mise en page de la section d'accroche principale .c (layout flex), .c1 / .c1a / .c1b (textes), .c1c a (bouton CTA), .c2 img (image)
Pourquoi : Le hero est spécifique à la page d'accueil. Les pages Portfolio ou Blog auront leur propre hero différent — ce fichier ne concerne que celui-ci.
services.css Section Mise en page de la section "Ce qu'on fait" .d (section wrapper), .d0 (titre), .dd (grille flex), .d1–.d4 (positionnement des cartes), .d1i–.d4i / .d1t–.d4t / .d1p–.d4p (contenu cartes)
Pourquoi : Ce fichier gère le layout et le contenu des cartes services. Les styles visuels des cartes (fond, border-radius) iront dans cards.css pour être réutilisés.
about.css Section Mise en page de la section "À propos" .e (layout flex 2 col), .e1 / .eimg (photo équipe), .e2 / .e2t / .e2p / .e2b a (texte + bouton)
Pourquoi : Cette section a une structure unique (image gauche + texte droite) qui ne se retrouve nulle part ailleurs. Elle mérite son propre fichier pour être modifiée indépendamment.
testimonials.css Section Mise en page de la section témoignages clients .f (section fond sombre), .f0 (titre blanc), .ff (grille flex), .f1–.f3 (positionnement), .f1t–.f3t / .f1n–.f3n (texte + auteur)
Pourquoi : La section témoignages a un fond sombre spécifique (#1a1a2e) qui la distingue visuellement. L'isoler permet de changer tout ce thème sombre en un seul fichier.
contact.css Section Mise en page et style du formulaire de contact .g / .g0 / .gf (section + titre + formulaire), .gi (inputs), .gt (textarea), .gb (bouton envoi + hover)
Pourquoi : Les styles de formulaire (inputs, textarea) sont spécifiques à cette section. Si demain on ajoute un formulaire sur la page Blog, il aura ses propres styles sans interférer.
footer.css Section Mise en page du pied de page .h (barre footer flex), .h1 (copyright), .h2 a / :hover (liens sociaux), .h3 (email)
Pourquoi : Comme le header, le footer est partagé entre toutes les pages. L'isoler permet de le mettre à jour globalement sans risquer de toucher à d'autres styles.
buttons.css Composant Styles de tous les boutons CTA du site Bouton rouge primaire (.c1c a, .gb), bouton navy secondaire (.e2b a), états hover
Pourquoi : Le site a 3 boutons avec des styles quasi-identiques éparpillés dans 3 fichiers différents. Les regrouper ici signifie qu'on change la couleur des boutons à un seul endroit.
cards.css Composant Styles visuels des cartes réutilisables Style de base des cartes services (.d1–.d4 : fond, padding, border-radius), style des cartes témoignages (.f1–.f3 : fond sombre, border-left accent)
Pourquoi : Les cartes services et témoignages partagent le même pattern visuel (flex:1, padding, border-radius). La page Portfolio en aura aussi — elle importera ce fichier directement.
04 — Stratégie d'évolution : les futures pages
📁 Page Portfolio
devstart/ ├── portfolio.html └── styles/ ├── base.css ← réutilisé ├── sections/ │ └── portfolio-hero.css ← nouveau │ └── portfolio-grid.css ← nouveau └── components/ └── cards.css ← réutilisé
La grille de projets réutilise cards.css. Seuls 2 nouveaux fichiers à créer.
📁 Page Blog
devstart/ ├── blog.html └── styles/ ├── base.css ← réutilisé ├── sections/ │ └── blog-list.css ← nouveau │ └── blog-article.css ← nouveau └── components/ └── cards.css ← réutilisé └── buttons.css ← réutilisé
Les articles de blog sont des cartes — cards.css est directement réutilisable.
📁 Espace Client
devstart/ ├── client.html └── styles/ ├── base.css ← réutilisé ├── sections/ │ └── client-dashboard.css ← nouveau │ └── client-sidebar.css ← nouveau └── components/ └── buttons.css ← réutilisé
L'espace client a une UI propre, mais les boutons et la base restent identiques.
📐 Règle générale d'extension
✓ Nouvelle page → nouveau dossier sections/ ✓ Nouveau composant → components/ ✓ base.css, buttons.css, cards.css → toujours réutilisés, jamais modifiés ✗ On ne touche jamais aux fichiers des autres pages pour une nouvelle page
Chaque nouvelle page est autonome et ne casse rien de ce qui existe.
05 — Ordre d'import dans index.html
/* Dans le <head> de index.html — l'ordre est important : global avant local */
<!-- 1. Fondations globales -->
<link rel="stylesheet" href="styles/base.css">
↑ Chargé en premier : reset + variables de couleur
 
<!-- 2. Composants réutilisables -->
<link rel="stylesheet" href="styles/components/buttons.css">
<link rel="stylesheet" href="styles/components/cards.css">
↑ Avant les sections car les sections s'appuient dessus
 
<!-- 3. Sections dans l'ordre d'apparition -->
<link rel="stylesheet" href="styles/sections/header.css">
<link rel="stylesheet" href="styles/sections/hero.css">
<link rel="stylesheet" href="styles/sections/services.css">
<link rel="stylesheet" href="styles/sections/about.css">
<link rel="stylesheet" href="styles/sections/testimonials.css">
<link rel="stylesheet" href="styles/sections/contact.css">
<link rel="stylesheet" href="styles/sections/footer.css">
✓ Conclusion de la Phase 2

L'architecture cible comprend 10 fichiers CSS répartis en 3 niveaux de responsabilité : 1 fichier global (base.css), 7 fichiers de sections (1 par bloc de page), et 2 fichiers de composants réutilisables (buttons.css, cards.css). Chaque fichier a une responsabilité unique et justifiée. Les futures pages (Portfolio, Blog, Espace Client) peuvent être ajoutées sans modifier un seul fichier existant — elles héritent des composants et créent uniquement ce qui leur est propre.
